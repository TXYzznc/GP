## Context

项目有121篇wiki文档，涉及配置表、系统设计、技术文档等多个领域。由于代码持续演进，大量文档出现字段不匹配、API接口过时、流程描述陈旧等问题。需要系统地对标代码实现逐一修正。

核心约束：
- 代码在 `Assets/AAAGame/Scripts/Game/` 及其子目录
- 配置表在 `Assets/AAAGame/DataTable/*.txt`（手工维护的源配置）
- 生成的代码在 `Assets/AAAGame/Scripts/Game/DataTable/*.cs`（自动生成，不手改）
- graphify知识图谱已部署，可快速查询代码依赖关系

## Goals / Non-Goals

**Goals:**
- 扫描所有121篇文档，识别过时内容
- 对每篇文档中提到的配置表字段、API、数值、流程逐一核验
- 使用graphify快速定位相关代码文件（而非逐个手动搜索）
- 更新文档使其与当前代码实现保持同步
- 标准化时间戳（所有改动文档更新为2026-04-17）
- 建立可追踪的进度管理（openspec3阶段制）

**Non-Goals:**
- 不重写或重构文档结构（只更新内容）
- 不创建新文档（仅更新已有wiki）
- 不涉及前端界面或新功能设计
- 不修改 `raw/` 或 `outputs/` 目录

## Decisions

### 决策1：分阶段验证与更新
- **第1阶段（配置表+技术文档）**：14文件，易于对标，风险低
- **第2阶段（UI+战斗系统）**：21文件，系统复杂，影响大
- **第3阶段（其他系统）**：86文件，架构相对稳定，可批量处理
- **理由**：优先处理高价值、高风险内容，逐步积累进度

### 决策2：graphify辅助验证流程
对每篇文档，工作流：
1. 读文档 → 识别核心系统和关键词（如"ItemManager"、"BuffTable"等）
2. 运行 `/graphify query "关键词"` 找到相关代码文件
3. 读代码 → 核验文档中的API、字段、流程
4. 对标配置表 → 检查数值、ID是否仍有效
5. 更新文档 → 修正不一致之处，添加`最后更新：2026-04-17`

- **理由**：避免无谓的全文搜索，大幅降低验证时间，提高准确度

### 决策3：并行验证多个系统
受限于token有限，单个会话无法完成所有121文件。采用multi-turn会话，每个会话处理一个阶段。

- **会话内并行**：在单次会话中，对同一子目录的多个文件并行读取（Reader Agent）
- **会话间续接**：openspec跟踪每个阶段的完成状态，下一会话从中断处继续
- **理由**：最大化吞吐量，避免重复验证

### 决策4：文档过时判断标准

| 文档类型 | 判断条件 | 验证来源 |
|---------|---------|---------|
| 配置指南 | 字段名/值与.txt配置表不符 | ItemTable.txt/BuffTable.txt等 |
| API文档 | 方法签名、参数名与.cs不符 | 对应Manager/Controller脚本 |
| 系统设计 | 流程、组件数量与实现不符 | 核心系统类（战斗/UI/背包等） |
| 问题诊断 | Bug是否已修复 | git log + 脚本当前状态 |
| 数值参考 | 数值表与公式与代码不符 | DamageCalculator、数值脚本 |

## Risks / Trade-offs

[风险1] 配置表更新频繁 → 需要持续维护
- **缓解**：建立标准化检查流程，文档更新时同步记录配置版本

[风险2] 代码跨度大（战斗/UI/背包等10+ 系统）→ 验证工作量大
- **缓解**：按优先级分阶段，每阶段聚焦一个主题，减轻认知负荷

[风险3] graphify查询可能存在假阴性 → 定位不到相关代码
- **缓解**：补充Grep手工搜索作为备案，保证完整性

[风险4] 多会话间状态丢失 → 重复验证或遗漏
- **缓解**：openspec完整记录每个阶段状态，提供resume机制

[权衡] 完整性 vs 效率
- 可选深度审查（核实每一行代码），或抽样检查（代表性文件）
- **建议**：初期采抽样检查，识别常见过时模式，再针对性修复

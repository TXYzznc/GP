# Design: Knowledge Base Refactoring

## Context

目前知识库包含 **108 个 markdown 文件**，分散在 **8 个子文件夹**中：
- 上下文缓存 (context backups)
- 开发总结 (development summaries)
- 开发注意事项 (development notes)
- 战斗与探索系统优化 (combat & exploration optimization)
- 技术文档 (technical docs)
- 系统设计 (system design)
- 问题诊断 (problem diagnosis)
- 项目规则 (project rules)

当前问题：
1. **缺乏统一模板**：不同类型文档格式不一致（开发总结、技术文档、系统设计）
2. **内容可能过时**：一边做一边写的文档，包含错误或过期信息
3. **冗余和重复**：可能存在重复的内容或冗余描述
4. **难以维护**：没有标准化的结构，新文档难以遵循约定

## Goals / Non-Goals

**Goals:**
- 为每种文档类型设计标准模板（5 种：系统设计、技术文档、开发总结、开发注意事项、问题诊断）
- 按照模板重构所有 108 个文件，确保内容准确和一致
- 提炼和精简文档内容，移除过时、重复或不必要的信息
- 建立清晰的文档维护规范，便于后续扩展

**Non-Goals:**
- 不修改代码或项目结构
- 不改变文档存储位置（保持现有 8 个文件夹）
- 不进行大规模的内容删除（整合而非删除）
- 不创建新的文件夹结构

## Decisions

### Decision 1: 创建 5 种标准模板
**选择**：为 5 种主要文档类型各创建一个模板
- **系统设计**：问题 → 方案 → 设计细节 → 实施步骤
- **技术文档**：概述 → 使用说明 → API/配置 → 示例 → 常见问题
- **开发总结**：时间范围 → 完成任务 → 技术要点 → 遗留问题
- **开发注意事项**：问题描述 → 解决方案 → 代码示例 → 相关链接
- **问题诊断**：症状 → 根本原因 → 修复方案 → 验证方法

**理由**：统一的模板能：
- 帮助读者快速找到需要的信息
- 让新文档作者知道应该写什么
- 便于自动化工具处理文档结构

### Decision 2: 分阶段重构（按文件夹）
**选择**：按优先级分阶段重构：
1. **优先级高**：项目规则、技术文档（最常使用）
2. **优先级中**：系统设计、开发总结
3. **优先级低**：开发注意事项、问题诊断、上下文缓存

**理由**：
- 高优先级文档影响更多人
- 可以先建立模板和流程，后续加快速度

### Decision 3: 保留元数据（日期、版本、状态）
**选择**：在每个文档的头部保留：
- 最后更新日期
- 文档状态（有效/过时/草稿）
- 关键版本标记（如 Phase 11-13）

**理由**：
- 帮助读者判断文档是否适用于当前项目状态
- 便于追踪文档的演变历史
- 快速识别需要更新的文档

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 重构过程中丢失信息 | 高 | 在重构前备份原始文件，逐个对比检查 |
| 新模板过于严格 | 中 | 设计灵活的模板，保留可选部分 |
| 工作量过大 | 高 | 分阶段进行，优先整理高价值文档 |
| 需要频繁更新模板 | 中 | 第一阶段后冻结模板，建立更新流程 |

## Migration Plan

### Phase 1: 模板设计与验证（1-2 天）
1. 为每种文档类型设计标准模板
2. 在 3-5 个现有文档上测试模板
3. 根据反馈调整模板

### Phase 2: 重构关键文档（3-5 天）
1. 重构项目规则（6 个文件）
2. 重构技术文档（45+ 个文件）
3. 验证结果并建立 checklist

### Phase 3: 重构其他文档（5-7 天）
1. 重构系统设计文档（40+ 个文件）
2. 重构开发总结文档（15+ 个文件）
3. 重构开发注意事项和问题诊断

### Phase 4: 最终审查和优化（2-3 天）
1. 全面审查所有重构文档
2. 更新 INDEX.md 和 README.md
3. 建立文档维护规范

## Open Questions

1. **过时文档处理**：是否应该将明确过时的文档移到专门的 archive 文件夹？
2. **文档审批**：重构后是否需要代码审查或技术审查？
3. **模板扩展**：未来如何添加新类型文档？
4. **维护频率**：建议多久审查和更新一次文档？

《诸神对决》项目架构设计方案 - Part 1：核心架构与基础设施
一、 总体架构设计理念
我们采用 ECS（Entity-Component-System）思想的混合架构 配合 Service Locator（服务定位器） 模式。

表现层 (View): Unity MonoBehaviour，只负责渲染和输入，不处理复杂逻辑。

逻辑层 (Controller/System): 纯 C# 类，处理战斗计算、AI决策、卡牌逻辑，便于单元测试和网络同步（Mirror）。


数据层 (Model): 强类型数据结构，负责序列化（二进制/云存档）。


核心分层图示
Infrastructure Layer (基础设施层): 资源加载、音频、输入、网络(Mirror)、事件总线。

Data Layer (数据层): 配置表(ScriptableObject)、存档数据(SaveData)。

Domain Layer (业务逻辑层): 召唤师系统、单位系统、背包系统、任务系统。

Presentation Layer (表现层): UI视图、场景特效、角色动画控制器。

二、 基础设施模块 (Infrastructure)
这部分脚本负责整个生命周期的管理，是所有功能的基石。

1. 全局管理类 (Game Entry)

类名: GameLifecycleManager (继承 NetworkManager - Mirror适配 )

作用: 游戏的入口，管理跨场景的单例，处理游戏整体状态机（启动 -> 基地 -> 探索 -> 战斗 -> 结算）。

主要属性:

GameState CurrentState (枚举: Base, Exploration, Combat, Extraction)

bool IsOnline (是否为PVP模式)

主要方法:

void InitializeGame(): 初始化所有Manager。

void SwitchScene(string sceneName): 场景切换处理（带Loading界面）。

void OnServerAddPlayer(): Mirror联机回调，处理玩家连接。

类名: EventManager

作用: 全局事件总线，解耦各个模块。

主要方法:

void AddListener<T>(Action<T> listener)

void RemoveListener<T>(Action<T> listener)

void TriggerEvent<T>(T eventData)

应用场景: 当“单位死亡”时发送事件，UI监听并更新血条，任务系统监听并计数。

2. 数据持久化与配置 (Data & Config)
针对策划案中提到的“二进制存储”和“云存档” 。


类名: SaveSystem

作用: 负责序列化和反序列化玩家数据。

主要方法:

void SaveGame(PlayerData data, bool toCloud = false): 保存到本地/云端。

PlayerData LoadGame(): 读取存档。

byte[] SerializeToBinary(object data): 二进制转换核心。

应用场景: 记录召唤师等级、拥有的神话卡牌、科技树解锁情况。

类名: ConfigManager

作用: 加载和缓存静态配置数据（Excel转Json或ScriptableObject）。

主要方法:

CardConfig GetCardConfig(int cardID): 获取策略卡数据。

《诸神对决》项目架构设计方案 - Part 2：核心玩法与系统实现
一、 探索与地图系统 (Exploration System)
根据策划案，探索是连接基地准备与战斗的中间环节，包含迷雾、资源收集和敌人遭遇机制 。

1. 地图与场景管理
类名: MapManager (Singleton)

作用: 管理当前探索地图的状态，处理战争迷雾和动态寻路。

主要属性:

GridSystem MapGrid: 用于管理地图格子、障碍物和迷雾数据。

List<MapEventPoint> ActiveEvents: 当前地图上存在的事件点（宝箱、商人等）。

主要方法:

void LoadMap(string mapId): 加载预设场景。


void RevealFog(Vector3 position, float radius): 玩家移动时消除迷雾 。


void BakeNavMeshDynamic(): 如果场景有动态变化，运行时重新烘焙寻路 。

类名: InteractionSystem

作用: 处理玩家与地图元素的交互（宝箱、NPC、陷阱）。

主要方法:

void TryInteract(IInteractable target): 尝试触发交互。

void SpawnEventReward(EventType type): 生成掉落物。

2. 敌人巡逻 AI (Overworld AI)
策划案明确指出了敌人具有“巡逻、警戒、追击、战斗、休息”五种状态 。这里使用 有限状态机 (FSM) 模式实现。

类名: EnemyOverworldController

作用: 控制大地图上敌人的行为逻辑。

属性:

EnemyState CurrentState: 当前状态。


float AlertValue: 警戒值 。


float PatrolRange: 巡逻/通信范围 。

主要方法:

void UpdateState(): 每帧运行状态逻辑。

void IncreaseAlert(float amount): 增加警戒值。


void BroadcastPosition(): 追击时向周围友军广播玩家位置 。

状态类设计 (State Pattern):

IEnemyState (接口): 定义 Enter(), Execute(), Exit()。

StatePatrol: 随机移动逻辑。


StateAlert: 原地静止，警戒条上涨逻辑 。


StateChase: 寻路追击玩家，离开活动区域后返回 。


StateRest: 静止回血，降低索敌范围 。


StateCombat: 检测到距离满足 R1，触发战斗场景切换 。

作用: 管理单场战斗的完整生命周期。

主要属性:

void OnUnitDeath(UnitEntity unit): 监听单位死亡，判断胜负。

void EndBattle(bool isWin): 结算战斗，处理经验和掉落。

五、 技术难点解决方案 (Technical Solutions)
1. 召唤物生命值继承机制 

实现方案:

在 MapManager 中维护一份 List<UnitRuntimeSnapshot>。

进入战斗前: 将 Snapshot 数据注入给 UnitEntity。

战斗结束时: 遍历存活的 UnitEntity，提取当前 HP，更新回 UnitRuntimeSnapshot。

死亡处理: 如果 HP <= 0，从 Snapshot 列表中移除该单位，标记为本局不可用。

2. 场景无缝切换与数据保留
实现方案:

使用 DontDestroyOnLoad 的 GameLifecycleManager 保存 SummonerRuntimeData 和 UnitRuntimeSnapshot。

探索 -> 战斗: 记录玩家在大地图的坐标 (Vector3)，加载战斗场景 (Scene Mode: Additive 或 Single)。

战斗 -> 探索: 卸载战斗场景，恢复玩家到之前记录的坐标，并根据战斗结果移除大地图上的敌人 GameObject。

架构图示化总结 (文本版)
GameLoop: GameLifecycleManager -> 状态机 (基地 -> 探索 -> 战斗 -> 结算)。

Input: InputManager -> SummonerController (移动) / CardManager (施法)。

Data Flow: SaveSystem <-> PlayerData <-> Inventory.

Events: EventManager 广播 OnEnemyKilled -> QuestSystem (计数) & LootManager (掉落)。

《诸神对决》项目架构设计方案 - Part 3：网络同步、美术管线与开发规划
一、 网络同步架构 (Networking with Mirror)
根据策划案 ，项目需要支持局域网/P2P联机，采用 Mirror 插件。为了支持“自走棋+实时卡牌”的同步，我们需要采用 Host-Client (主机-客户端) 模式。

1. 网络核心组件
类名: GameNetworkManager (继承 NetworkManager)

作用: 扩展 Mirror 的基础管理器，处理玩家连接、断开、以及场景切换时的网络对象生成。

主要方法:

OnServerAddPlayer(): 当玩家连接时，生成 SummonerPlayer 对象（玩家替身）。

OnServerDisconnect(): 处理掉线（如：判定对方胜利）。

StartHostPVE(): 启动单人/合作模式（本机作为 Host）。

StartClientPVP(): 加入局域网对战。

类名: NetworkTimeSync


作用: 策划案提到“校验时间戳防止作弊” 以及“Lockstep机制” 。对于毕设，建议使用 状态同步 (State Synchronization) 配合 确定性随机数。


主要方法:

SyncRNGSeed(int seed): 在战斗开始时，Host 生成一个随机种子并通过 [ClientRpc] 发送给 Client。双方使用该种子初始化各自的 Random 实例，确保自动战斗中“暴击”、“闪避”等概率结果一致。

2. 玩家操作同步 (Player Logic)
类名: SummonerNetworkController (继承 NetworkBehaviour)

作用: 处理玩家的手动输入（放卡、移动、技能）并同步给服务器。

主要属性:

[SyncVar] int CurrentMana: 同步灵力值，防作弊。

[SyncVar] int CurrentHP: 同步血量。

主要方法:

[Command] void CmdCastSpell(int cardID, Vector3 target): 客户端调用，服务器执行。服务器验证灵力是否足够，若足够则扣除灵力并生成法术效果。

[ClientRpc] void RpcOnSpellCast(int cardID, Vector3 target): 服务器调用，客户端执行。在所有客户端播放法术特效。

[Command] void CmdDeployUnit(int unitID, int gridIndex): 请求部署棋子。

3. 战斗单位同步 (Unit Logic)
由于自走棋单位较多，如果每个单位都实时同步 Transform 会造成带宽压力。

优化方案: 仅同步关键状态和关键事件，位置计算依赖本地导航网格（NavMesh）+ 相同的 RNG 种子。

类名: UnitNetworkView (继承 NetworkBehaviour)

作用: 挂载在棋子 Prefab 上。

主要方法:

[SyncVar] float NetworkHP: 同步血量（以服务器为准）。

[ClientRpc] void RpcTriggerAnimation(string triggerName): 同步攻击/技能动作。

[ClientRpc] void RpcSyncPosition(Vector3 pos): 低频率（如每秒1次）校准位置，防止两端误差过大。

二、 美术与表现管线 (Art Pipeline)
针对策划案中提出的“三渲二”、“非人角色动画” 、“天气系统” 和 “演出系统” 。



1. 动画系统解决方案
针对你提出的 “非人角色怎么绑定骨骼和制作动画？”  的问题：

解决方案: Unity 提供了 Generic 动画类型，专门用于非人型骨骼（四足、异形）。

类名: UnitAnimationController

作用: 统一管理 Animator。

设计: 使用 Animator Override Controller。所有棋子共用一套基础状态机（Idle -> Run -> Attack -> Death），不同角色只需替换具体的 Animation Clip（动画片段）。

主要方法:

void SetSpeed(float speed): 控制移动动画混合。

void TriggerAttack(float attackSpeedMultiplier): 根据攻速调整攻击动画播放速度。

2. 天气与环境系统
类名: WeatherManager


作用: 实现昼夜交替和天气变化 。

主要属性:

Volume GlobalPostProcessing: 引用后处理体积（Bloom, Color Grading）。

Light MainDirectionalLight: 引用主光源（太阳/月亮）。

ParticleSystem RainSystem: 雨雪粒子。

主要方法:

void SetWeather(WeatherType type): 切换天气（如：雨天 -> 降低光照强度，开启雨粒子，增加地面材质的光滑度）。

void UpdateDayNightCycle(float gameTime): 根据游戏时间旋转光源，改变天空盒颜色。

3. 演出与剧情系统
类名: CutsceneDirector


作用: 管理 Unity Timeline 和视频播放 。

主要方法:

void PlayTimeline(string timelineId): 播放 BOSS 出场或剧情对话。

void PauseGameLogic(): 演出时暂停战斗逻辑（但保持动画播放）。

事件联动: 结合 EventManager，当玩家第一次进入某地图时 (OnMapEnter)，自动触发开场 Timeline。

四、 毕设开发路线图 (Development Roadmap)
基于毕设的时间压力，建议采用 MVP (Minimum Viable Product，最小可行性产品) 迭代开发法。

阶段 1：核心原型 (Core Prototype) - 预计耗时：2-3周
目标: 实现单机版“放置棋子 -> 自动战斗 -> 结算”。

任务:

搭建基础架构 (GameLifecycle, EventManager)。

实现 UnitEntity 和基础 AI（自动寻敌、攻击）。

实现简单的 UI（显示血条）。

产出: 一个灰盒场景，并在场景中可以自动打死敌人。

阶段 2：循环打通 (Game Loop) - 预计耗时：3-4周
目标: 完成“基地 -> 地图探索 -> 战斗 -> 撤离”循环。

任务:

实现 MapManager 和战争迷雾。

实现 Inventory 和 LootManager（搜打撤）。

接入 SaveSystem，实现数据在场景间的传递（生命继承 ）。

实现卡牌系统 CardManager（手动释放技能）。

阶段 3：内容填充与表现 (Content & Art) - 预计耗时：3-4周
目标: 替换美术资源，增加角色和职业。

任务:

导入模型，配置 AnimationController。

实现 SummonerSkillSystem（职业技能树 ）。

添加特效（三渲二材质、技能特效）。

实现天气系统和简单的 Timeline 演出。

阶段 4：联机与优化 (Networking & Polish) - 预计耗时：2-3周

目标: 接入 Mirror，实现局域网对战 。

任务:

将单机逻辑改造为 Command/Rpc。

实现 NetworkRoom (房间匹配)。

Debug 同步问题。

注意: 如果时间紧迫，优先保证前三个阶段的单机体验完美，联机作为加分项。

五、 总结与建议
这份架构设计旨在平衡 “策划案的宏大需求” 与 “毕设的落地可行性”。

架构核心: 利用 ScriptableObject 驱动数据，利用 EventManager 解耦逻辑，利用 FSM 管理复杂的 AI 状态。

难点攻克:

生命继承: 利用全局单例 RuntimeData 在场景切换时暂存数据。

联机同步: 优先做状态同步，不要死磕帧同步（Lockstep），因为 Unity 物理引擎本身具有不确定性。

非人动画: 善用 Unity 的 Generic Rig 和 Asset Store 资源。

# AI 开发管线问题清单 (2026-04-17)

## 📋 概述

基于对项目 CLAUDE.md、AGENTS.md、settings.json 和知识库的完整审视，识别出 4 个主要问题和若干优化机会。整体评价：**基础设施强、自动化弱、流程未优化**。

---

## 🚧 主要问题（4个）

### 问题1️⃣：自动化不足 — 工作流中断点多

**严重程度：** 🔴 高 | **影响范围：** 日常开发流程

**现状：**
- `settings.json` 极简，没有任何 hook 配置
- DataTableGenerator 需要手动运行（每次配置表修改后）
- 没有预 commit 检查机制
- 没有自动化脚本生成或资源导入工作流

**具体表现：**
```
开发流程中的手动干预点：
1. 修改配置表 Excel
2. ❌ 手动运行 DataTableGenerator → 生成 .cs + .bytes
3. 修改 C# 脚本
4. ❌ 手动提交前检查（代码规范、配置表同步）
5. 生成新卡牌时需要：
   - ❌ 手动调用外部图片 API
   - ❌ 手动放置资源到项目
   - ❌ 手动设置 Unity 资源导入参数
```

**根本原因：**
- settings.json 没有配置 hooks（harness 机制未启用）
- 缺少关键步骤的自动化脚本

**影响：**
- ⏱️ 每个配置表修改都多出 2-3 分钟的手动操作
- 🐛 容易忘记运行 DataTableGenerator，导致代码生成文件过期
- 🔄 无法跟踪哪些配置表已同步、哪些未同步

**优先级：** ⭐⭐⭐ 高（立即改进）

---

### 问题2️⃣：Multi-Agent 协作流程缺失

**严重程度：** 🟡 中 | **影响范围：** 复杂任务效率

**现状：**
- 没有标准化的多 Agent 并行协作模板
- 没有 Agent 间数据传递的最佳实践文档
- CLAUDE.md 中的 5 步工作流是**顺序执行**，没有考虑并行可能性

**具体表现 - 以卡牌设计为例：**
```
当前（顺序执行）：
Agent-1 分析卡牌系统架构 → 5分钟
Agent-2 设计画面效果提示词 → 5分钟
Agent-3 搜索相似卡牌实现 → 5分钟
==================== 总计 15分钟

可能（并行执行）：
Agent-1、Agent-2、Agent-3 同时运行
==================== 总计 5分钟
```

**根本原因：**
- 缺少"如何在并行 Agent 间传递上下文"的文档示例
- CLAUDE.md 中的工作流没有标记可并行的步骤
- 没有"Agent 结果综合"的标准模板

**影响：**
- ⏱️ 复杂任务执行时间翻倍
- 🧠 每次都要临时设计协作策略，不规范
- 📊 无法衡量并行收益

**优先级：** ⭐⭐ 中（下周改进）

---

### 问题3️⃣：专项 Skill 不够细化

**严重程度：** 🟡 中 | **影响范围：** 特定工作流的易用性

**现状：**
- 现有 Skill（gitnexus、ai-art、dev-tools）是**通用工具**
- 缺少针对**项目特定业务流程**的 Skill
- 每次执行"卡牌设计""Buff实现""UI重构"时，都需要从 CLAUDE.md 推导工作流

**缺失的 Skill：**
```
当前有：                      需要补充：
✅ gitnexus-*                 ❌ card-design-workflow
✅ ai-art                     ❌ buff-system-implementation  
✅ dev-tools                  ❌ ui-rebuild-workflow
✅ openspec                   ❌ enemy-spawn-integration
✅ unity-dev                  ❌ config-table-maintenance
```

**具体表现：**
- 要做卡牌设计，需要自己拼凑：
  1. 读 INDEX.md 找相关文档
  2. 读 CLAUDE.md 推导工作流
  3. 手动协调 ai-art + 外部 API + 脚本生成
  
- 而一个`card-design-workflow` Skill 就能一键启动整个流程

**根本原因：**
- 项目初期没有预见这些高频业务流程
- Skill 还停留在"通用工具"阶段，没有"领域工作流"

**影响：**
- 🔍 每次都要搜索、思考、拼凑，浪费认知负荷
- 📚 新开发者学习曲线陡峭
- 🔄 无法标准化每个流程的最佳实践

**优先级：** ⭐⭐ 中（2周内补充前3个）

### 问题4️⃣：内存系统利用率低

**严重程度：** 🟡 中 | **影响范围：** 长期知识积累和重用

**现状：**
- MEMORY.md 只有 3 条记录
- 关键架构决策、常见 bug 修复方案、性能优化经验**没有沉淀**
- 每次遇到类似问题都要重新分析

**具体表现：**
```
当前的 MEMORY.md：
- container_architecture_status.md（容器系统修复状态）
- progress_overview_2026_04_03.md（项目进度）
- damage_calculation_design.md（伤害计算）

缺失的：
- 各系统的核心设计决策及 Why
- 常见 bug 修复方案（Buff、UI、战斗等）
- 性能优化经验汇总
- 与外部 API 的集成经验
- DataTable 迁移教训
- UI 坐标系统的陷阱记录
```

**根本原因：**
- 没有建立"项目知识积累"的制度化流程
- 没有明确何时应该写 memory 记录

**影响：**
- 🔄 重复工作：同样的 bug 调试多次
- 🧠 认知负荷高：每次都要重新推导而非查询
- 📈 学习曲线平缓：无法快速积累项目经验

**优先级：** ⭐ 低（持续改进）

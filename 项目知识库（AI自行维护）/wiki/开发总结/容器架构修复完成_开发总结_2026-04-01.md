## 🔴 修复的 Bug

**原因**: 仓库→仓库的 SwapSlots 已经完成了交换，但之后 RemoveItem() 又被调用，导致数据重复修改

**修复**:
```csharp
// 之前（错误）
if (success)
{
    m_WarehouseManager.RemoveItem(fromSlotIndex, count);
}

// 之后（正确）
if (success && targetContainer is not WarehouseSlotContainerImpl)
{
    m_WarehouseManager.RemoveItem(fromSlotIndex, count);
}
```

           otherContainerType == SlotContainerType.FastBar;

## 🔍 架构设计总结

### 核心原则（修复后）

1. **单一责任**: 每个 MoveToXXX 方法只处理"目标"相关逻辑，源格子清理由 ExecuteMove 统一处理
2. **对称性**: 相同容器之间（仓库↔仓库、快捷栏↔快捷栏）支持交换（通过 SwapSlots）；不同容器之间只支持单向存入
3. **数据一致性**: 避免在 MoveToXXX 中修改源格子后再在 ExecuteMove 中修改，防止数据竞争

### 设计权衡

| 权衡项 | 选择 | 理由 |
|--------|------|------|
| **交换支持** | 仅在同容器内 | 避免跨容器交换的复杂性和数据竞争 |
| **仓库↔快捷栏** | 禁止 | 快捷栏应通过背包中转，更符合游戏设计 |
| **错误处理** | 操作失败返回 false | 保留用户灵活操作的空间（可以清空后重试） |

## 一、架构概览

### ISlotContainer 系统层次

```
ISlotContainer（接口）
    ↓
SlotContainerBase（抽象基类）
    ├─ InventorySlotContainerImpl（背包容器）
    ├─ WarehouseSlotContainerImpl（仓库容器）
    ├─ FastBarSlotContainerImpl（快捷栏容器）
    └─ EquipSlotContainerImpl（装备栏容器框架）
```

### 核心设计原则

1. **单一责任**: MoveToXXX 方法只处理目标相关逻辑，源格子清理由 ExecuteMove 统一处理
2. **对称性**: 容器内操作支持交换（SwapSlots）；跨容器操作只支持存入/堆叠
3. **数据一致性**: 避免重复操作，清空源格子基于目标格子状态判断

### 清空源格子的统一条件

```csharp
if (success && targetContainer is not [SameContainerType] &&
    (targetIsEmpty || targetSlot.ItemId == itemId))
{
    ClearSource();
}
```

| # | 位置 | 问题 | 严重程度 |
|---|------|------|---------|
| 1 | WarehouseSlotContainerImpl.cs:63-66 | 仓库内 SwapSlots 完成后又调用 RemoveItem，导致重复修改 | P0 数据丢失 |
| 2 | FastBarSlotContainerImpl.cs:52-55 | 快捷栏内 SwapSlots 完成后又调用 ClearSlot，刚交换的物品被清空 | P0 数据丢失 |

**问题根源**：原设计用容器类型决定是否清空源格子，实际上应用目标格子状态决定——这是更清晰灵活的方式。

### Bug 3：快捷栏↔仓库不合理交互

**设计理由**: 快捷栏是"背包的快速访问栏"，物品流应通过背包中转，不应直接与仓库交互。

**FastBarSlotContainerImpl.cs**:
```csharp
// 修复后（删除 Warehouse 选项）
public override bool CanInteractWith(SlotContainerType otherContainerType)
    => otherContainerType == SlotContainerType.Inventory ||
       otherContainerType == SlotContainerType.FastBar;
```

**WarehouseSlotContainerImpl.cs**:
```csharp
// 修复后（删除 FastBar 选项）
public override bool CanInteractWith(SlotContainerType otherContainerType)
    => otherContainerType == SlotContainerType.Inventory ||
       otherContainerType == SlotContainerType.Warehouse;
```

### 设计改进：简化跨容器交换逻辑

**仓库→背包** 和 **快捷栏→背包** 的 MoveToInventory 方法原来有复杂的"目标物品回源容器"交换逻辑，现在简化为：
- 目标格子为空 → 存入
- 目标格子同种物品 → 堆叠
- 目标格子有不同物品 → **操作失败**（用户可先清空再操作）

**理由**: 交换逻辑会导致数据竞争（RemoveItem 会操作新物品而非源物品）。简化后逻辑清晰，不影响用户体验。

## 四、操作规则总结（修复后）

| 源 ← → 目标 | 允许 | 行为 |
|----------|------|------|
| 背包 ↔ 背包 | ✓ | 存入、堆叠、**交换** |
| 背包 ↔ 仓库 | ✓ | 存入、堆叠（不支持交换） |
| 背包 ↔ 快捷栏 | ✓ | 存入、堆叠（不支持交换） |
| 仓库 ↔ 仓库 | ✓ | 存入、堆叠、**交换** |
| 仓库 ↔ 快捷栏 | **✗** | 已禁止（违反设计） |
| 快捷栏 ↔ 快捷栏 | ✓ | 存入、堆叠、**交换** |

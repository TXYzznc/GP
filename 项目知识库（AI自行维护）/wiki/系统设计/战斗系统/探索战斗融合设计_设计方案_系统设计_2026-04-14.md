游戏的战斗阶段设计（肉鸽+自走棋+卡牌）融合顺利，但探索阶段的"搜打撤"玩法与战斗核心玩法存在融合矛盾：

1. **探索状态和战斗状态严格分离**
   - 玩家在探索时无法攻击敌人
   - 必须进入战斗状态才能对战

2. **玩家缺少主动权**
   - 敌人发现玩家后，玩家只能逃跑
   - 无法反制或偷袭敌人

3. **缺少传统搜打撤的核心体验**
   - 传统搜打撤：玩家可以在探索时造成伤害
   - 当前设计：探索和战斗完全隔离

4. **策略深度不足**
   - 玩家的选择被限制
   - 缺少紧张刺激感

---

## 解决方案

### 方案一：遭遇战机制（推荐）⭐⭐⭐

#### 核心思路

在探索状态下允许玩家主动发起"快速战斗"，但保留完整战斗的策略深度。

#### 设计要点

**1. 主动攻击触发器**
- 玩家在探索状态下可以对敌人使用"召唤师技能"进行攻击
- 攻击命中后，立即触发战斗状态切换
- 如果玩家先手攻击，进入战斗时获得"先手优势"Buff

**2. 偷袭机制**
- 当敌人处于"休息状态"时，玩家可以进行偷袭
- 偷袭成功：敌人进入战斗时血量降低20-30%，或者被眩晕1-2回合
- 偷袭失败：正常进入战斗

**3. 快速脱战机制**
- 战斗开始后，如果玩家在前3秒内选择"撤退"，可以快速脱战
- 脱战成功率与敌人类型相关（普通敌人90%，精英70%，Boss 30%）
- 脱战失败：继续战斗，但召唤师受到一定伤害

#### 实现示例

```csharp
// 探索状态下的攻击系统
public class ExplorationCombatSystem
{
    // 玩家在探索状态下使用召唤师技能攻击敌人
    public void PlayerAttackEnemy(EnemyEntity enemy)
    {
        // 1. 判断敌人状态
        if (enemy.CurrentState == EnemyState.Rest)
        {
            // 偷袭成功
            enemy.ApplyDebuff(DebuffType.Ambushed, 0.3f); // 血量-30%
            ShowFloatingText("偷袭成功！");
        }
        
        // 2. 给玩家先手优势
        PlayerBuffManager.AddBuff(BuffType.FirstStrike, 10f); // 攻击力+10%，持续10秒
        
        // 3. 触发战斗
        GameStateManager.Instance.SwitchToCombat(enemy);
    }
    
    // 快速脱战机制
    public bool TryQuickRetreat(float elapsedTime)
    {
        if (elapsedTime > 3f) return false; // 超过3秒无法快速脱战
        
        float successRate = CalculateRetreatRate();
        if (Random.value < successRate)
        {
            // 脱战成功
            GameStateManager.Instance.SwitchToExploration();
            return true;
        }
        else
        {
            // 脱战失败，受到惩罚
            PlayerAttribute.TakeDamage(50f);
            return false;
        }
    }
}
```

#### 优缺点分析

**优点：**
- 保留了探索和战斗的状态分离，技术架构不需要大改
- 增加了玩家的主动权和策略选择
- 实现了偷袭机制，提升了搜打撤的体验
- 快速脱战机制给了玩家"试探"的机会

**缺点：**
- 需要新增探索状态下的攻击系统
- 需要设计先手优势和偷袭的数值平衡

### 方案二：混合战斗模式（中等推荐）⭐⭐

#### 核心思路

探索状态下允许"轻量级战斗"，只有进入正式战斗才切换到战斗场景。

#### 设计要点

**1. 探索战斗模式**
- 玩家在探索状态下可以召唤1-2个棋子进行战斗
- 战斗在探索场景中进行，不切换场景
- 只能使用基础攻击和召唤师技能，不能使用策略卡

**2. 正式战斗触发条件**
- 当敌人血量降低到50%以下时，触发"激怒"状态
- 激怒状态下，敌人会呼叫增援，强制进入正式战斗
- 或者玩家可以选择"全力以赴"，主动进入正式战斗

**3. 战斗奖励差异**
- 探索战斗击败敌人：获得50%奖励
- 正式战斗击败敌人：获得100%奖励
- 鼓励玩家进入正式战斗，但也给了"快速清理小怪"的选项

#### 实现示例

```csharp
// 混合战斗系统
public class HybridCombatSystem
{
    private bool m_IsLightCombat = false; // 是否为轻量级战斗
    
    // 开始轻量级战斗
    public void StartLightCombat(EnemyEntity enemy)
    {
        m_IsLightCombat = true;
        
        // 1. 不切换场景，在探索场景中战斗
        // 2. 只允许召唤1-2个棋子
        // 3. 禁用策略卡系统
        
        // 监听敌人血量
        enemy.OnHealthChanged += CheckEnrageCondition;
    }
    
    // 检查激怒条件
    private void CheckEnrageCondition(float currentHP, float maxHP)
    {
        if (currentHP / maxHP < 0.5f)
        {
            // 敌人激怒，强制进入正式战斗
            ShowWarning("敌人呼叫增援！");
            TransitionToFullCombat();
        }
    }
    
    // 转换为正式战斗
    private void TransitionToFullCombat()
    {
        m_IsLightCombat = false;
        
        // 1. 切换到战斗场景
        // 2. 保留当前战斗状态（血量、位置等）
        // 3. 解锁完整战斗功能
        
        GameStateManager.Instance.SwitchToCombat();
    }
}
```

#### 优缺点分析

**优点：**
- 给了玩家"快速清理小怪"的选项
- 保留了正式战斗的策略深度
- 增加了战斗的层次感

**缺点：**
- 需要实现两套战斗系统（轻量级和完整版）
- 技术复杂度较高
- 可能导致玩家混淆

### 方案三：警戒阶段互动（轻量推荐）⭐

#### 核心思路

在敌人的"警戒状态"下增加玩家的互动选项。

#### 设计要点

**1. 警戒阶段的选择**

当敌人进入警戒状态时，玩家可以选择：
- **逃跑**：离开警戒范围，警戒值下降
- **偷袭**：消耗圣水进行偷袭，成功则进入战斗时获得优势
- **谈判**：消耗金币贿赂敌人，避免战斗（仅对部分敌人有效）

**2. 偷袭判定**
- 偷袭成功率与玩家等级、敌人类型相关
- 偷袭失败：立即进入战斗，且敌人获得"警觉"Buff

**3. 谈判系统**
- 普通敌人：可以贿赂
- 精英敌人：贿赂成功率50%
- Boss：无法贿赂

#### 实现示例

```csharp
// 警戒阶段互动系统
public class AlertPhaseInteraction
{
    // 显示互动选项
    public void ShowInteractionOptions(EnemyEntity enemy)
    {
        // 显示UI选项
        UIManager.ShowAlertInteractionUI(new InteractionOption[]
        {
            new InteractionOption("逃跑", () => Flee(enemy)),
            new InteractionOption("偷袭", () => Ambush(enemy)),
            new InteractionOption("贿赂", () => Bribe(enemy), enemy.CanBeBribed)
        });
    }
    
    // 偷袭
    private void Ambush(EnemyEntity enemy)
    {
        if (!PlayerInventory.HasHolyWater(1))
        {
            ShowMessage("圣水不足！");
            return;
        }
        
        PlayerInventory.ConsumeHolyWater(1);
        
        float successRate = CalculateAmbushRate(enemy);
        if (Random.value < successRate)
        {
            // 偷袭成功
            enemy.ApplyDebuff(DebuffType.Ambushed, 0.2f);
            EnterCombat(enemy, hasAdvantage: true);
        }
        else
        {
            // 偷袭失败
            enemy.ApplyBuff(BuffType.Alert, 10f);
            EnterCombat(enemy, hasAdvantage: false);
        }
    }
}
```

#### 优缺点分析

**优点：**
- 实现简单，不需要大改架构
- 增加了策略选择
- 利用了现有的警戒状态机制

**缺点：**
- 互动深度有限
- 仍然无法在探索状态下直接战斗

### 方案四：召唤师形态系统（创新方案）⭐⭐⭐⭐

#### 核心思路

将探索状态和战斗状态的切换，重新定义为"召唤师形态"的切换。

#### 设计要点

**1. 探索形态（人形态）**
- 玩家可以使用召唤师技能进行攻击
- 可以召唤1个"护卫棋子"跟随
- 攻击力较弱，但移动速度快

**2. 战斗形态（召唤师形态）**
- 进入战斗场景，解锁完整战斗能力
- 可以召唤多个棋子，使用策略卡
- 攻击力强，但无法移动（自走棋机制）

**3. 形态切换**
- 玩家可以主动切换形态（冷却时间30秒）
- 在探索形态下攻击敌人，会自动切换到战斗形态
- 战斗结束后，自动切换回探索形态

**4. 护卫棋子机制**
- 探索形态下，玩家可以召唤1个护卫棋子
- 护卫棋子可以攻击敌人，但战斗力有限
- 护卫棋子的血量在整局游戏中继承

#### 实现示例

```csharp
// 召唤师形态系统
public class SummonerFormSystem
{
    public enum FormType
    {
        Exploration,  // 探索形态
        Combat        // 战斗形态
    }
    
    private FormType m_CurrentForm = FormType.Exploration;
    private ChessEntity m_GuardChess; // 护卫棋子
    
    // 切换形态
    public void SwitchForm(FormType targetForm)
    {
        if (m_CurrentForm == targetForm) return;
        
        m_CurrentForm = targetForm;
        
        if (targetForm == FormType.Combat)
        {
            // 进入战斗形态
            GameStateManager.Instance.SwitchToCombat();
            
            // 护卫棋子自动加入战斗
            if (m_GuardChess != null)
            {
                CombatChessManager.AddChess(m_GuardChess);
            }
        }
        else
        {
            // 进入探索形态
            GameStateManager.Instance.SwitchToExploration();
            
            // 保留一个护卫棋子
            m_GuardChess = CombatChessManager.GetStrongestChess();
        }
    }
    
    // 探索形态下的攻击
    public void ExplorationAttack(EnemyEntity enemy)
    {
        // 1. 使用召唤师技能攻击
        PlayerSkillManager.UseSkill(0, enemy.transform.position);
        
        // 2. 护卫棋子攻击
        if (m_GuardChess != null)
        {
            m_GuardChess.Attack(enemy);
        }
        
        // 3. 如果敌人血量低于50%，自动切换到战斗形态
        if (enemy.HealthPercent < 0.5f)
        {
            SwitchForm(FormType.Combat);
        }
    }
}
```

#### 优缺点分析

**优点：**
- 符合游戏的世界观设定（召唤师形态）
- 给了玩家在探索状态下的战斗能力
- 保留了完整战斗的策略深度
- 护卫棋子机制增加了探索阶段的陪伴感

**缺点：**
- 需要重新设计召唤师的能力体系
- 需要平衡两种形态的战斗力

## 推荐方案：探索战斗混合系统

### 核心设计

结合**方案一（遭遇战机制）+ 方案四（召唤师形态）**，打造完整的探索战斗混合系统。

### 系统架构

#### 1. 探索形态

**基础能力：**
- 玩家可以召唤1个护卫棋子跟随
- 可以使用召唤师技能攻击敌人
- 攻击敌人后，触发"遭遇战"

**护卫棋子特性：**
- 从已拥有的棋子中选择1个作为护卫
- 护卫棋子战斗力为完整战斗时的70%
- 血量在整局游戏中继承
- 可以随时更换护卫棋子（在非战斗状态）

#### 2. 遭遇战机制

**偷袭休息中的敌人：**
- 敌人血量-30%
- 敌人被眩晕1回合
- 玩家获得"先手优势"Buff（攻击力+10%，持续10秒）
- 进入战斗

**主动攻击警戒中的敌人：**
- 玩家获得"先手优势"Buff
- 正常进入战斗

**被敌人发现：**
- 正常进入战斗
- 无额外优势

#### 3. 快速脱战机制

**触发条件：**
- 战斗开始后3秒内
- 玩家选择"撤退"按钮

**成功率：**
- 普通敌人：90%
- 精英敌人：70%
- Boss：30%

**脱战成功：**
- 立即返回探索状态
- 敌人恢复到战斗前的状态
- 玩家无惩罚

**脱战失败：**
- 继续战斗
- 召唤师受到50点固定伤害
- 敌人获得"警觉"Buff（攻击力+5%）

#### 4. 战斗形态

**完整战斗能力：**
- 进入完整的自走棋战斗场景
- 解锁所有棋子和策略卡
- 护卫棋子自动加入战斗阵容
- 使用完整的战斗规则

**战斗结束：**
- 自动切换回探索形态
- 可以选择一个棋子作为新的护卫
- 继承战斗后的血量状态

```csharp
// 护卫棋子管理器
public class GuardChessManager : MonoBehaviour
{
    private ChessEntity m_CurrentGuard;
    
    // 设置护卫棋子
    public void SetGuardChess(ChessEntity chess)
    {
        if (m_CurrentGuard != null)
        {
            // 移除旧护卫
            m_CurrentGuard.SetAsGuard(false);
        }
        
        m_CurrentGuard = chess;
        m_CurrentGuard.SetAsGuard(true);
        
        DebugEx.Log("GuardChessManager", $"设置护卫棋子: {chess.Config.Name}");
    }
    
    // 获取护卫棋子
    public ChessEntity GetGuardChess()
    {
        return m_CurrentGuard;
    }
}
```

```csharp
// 先手优势Buff
public class FirstStrikeBuff : BuffBase
{
    private float m_AttackBonus = 0.1f; // 10%攻击力加成
    
    public override void OnApply(ChessEntity target)
    {
        base.OnApply(target);
        
        // 增加攻击力
        target.Attribute.ModifyAttack(m_AttackBonus, true);
        
        DebugEx.Success("FirstStrikeBuff", $"先手优势生效: 攻击力+{m_AttackBonus * 100}%");
    }
    
    public override void OnRemove(ChessEntity target)
    {
        // 移除攻击力加成
        target.Attribute.ModifyAttack(-m_AttackBonus, true);
        
        base.OnRemove(target);
    }
}
```

```csharp
// 快速脱战UI
public class QuickRetreatUI : UIFormBase
{

    [SerializeField] private Text m_TimerText;
    
    private float m_TimeRemaining = 3f;

        base.OnOpen(userData);
        
        m_RetreatButton.onClick.AddListener(OnRetreatClicked);
        m_TimeRemaining = 3f;
    }
    
    protected override void OnUpdate(float elapseSeconds, float realElapseSeconds)
    {
        base.OnUpdate(elapseSeconds, realElapseSeconds);
        
        m_TimeRemaining -= elapseSeconds;
        m_TimerText.text = $"撤退时间: {m_TimeRemaining:F1}s";
        
        if (m_TimeRemaining <= 0)
        {
            // 时间到，隐藏按钮
            m_RetreatButton.gameObject.SetActive(false);
        }
    }
    
    private void OnRetreatClicked()
    {
        // 尝试快速脱战
        ExplorationCombatSystem.Instance.TryQuickRetreat();
    }
}
```

## 总结

通过引入**探索战斗混合系统**，我们成功解决了探索阶段与战斗核心玩法的融合问题：

1. **保留了技术架构**：探索和战斗状态仍然分离，不需要大规模重构
2. **增加了玩家主动权**：可以主动攻击、偷袭、快速脱战
3. **提升了策略深度**：护卫棋子选择、偷袭时机、脱战判断
4. **符合世界观设定**：召唤师形态的概念自然融入游戏
5. **保持了搜打撤体验**：偷袭、试探、撤退等核心玩法得以实现

这个方案既解决了当前的设计矛盾，又为未来的扩展留下了空间，是一个平衡了技术可行性和游戏体验的优质方案。

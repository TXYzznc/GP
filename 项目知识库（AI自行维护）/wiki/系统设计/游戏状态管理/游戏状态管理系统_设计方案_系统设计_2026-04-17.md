## 一、设计原则

### 1. 轻量化设计
- 避免过度封装和不必要的抽象层
- 减少冗余的校验方法
- 直接使用 GF 框架的 FSM，不额外包装

### 2. 低耦合设计
- 状态之间通过事件通信，不直接引用
- UI 通过事件监听状态变化，自动显示/隐藏
- 系统模块独立，可单独测试

### 3. 灵活扩展
- 新增状态只需添加新的 State 类
- UI 模块可独立开发和替换
- 配置驱动的状态切换规则

## 二、核心架构

### 1. 状态机结构

```
GameStateManager (单例)
├── 主状态机 (IFsm<GameStateManager>)
│   ├── OutOfGameState (局外状态)
│   └── InGameState (局内状态)
│       └── 子状态机 (IFsm<InGameState>)
│           ├── ExplorationState (探索状态)
│           └── CombatState (战斗状态)
```

### 2. 状态类型定义

```csharp
// 主状态类型
public enum GameStateType
{
    OutOfGame,  // 局外
    InGame      // 局内
}

// 局内子状态类型
public enum InGameStateType
{
    Exploration,  // 探索
    Combat        // 战斗
}
```

## 三、核心脚本设计

### 1. GameStateManager（游戏状态管理器）

**职责**：
- 管理主状态机
- 提供状态切换接口
- 作为状态机的 Owner

**核心方法**：
```csharp
// 切换到局外
public void SwitchToOutOfGame()

// 切换到局内
public void SwitchToInGame()

// 获取当前主状态
public GameStateType CurrentState

// 获取当前局内子状态（如果在局内）
public InGameStateType? CurrentInGameState
```

**设计要点**：
- 单例模式，全局唯一
- 不做复杂的状态校验，直接切换
- 状态切换失败由 GF 的 FSM 自动处理

[↑ 返回目录](#目录)

### 2. OutOfGameState（局外状态）

**职责**：
- 加载局外场景
- 触发局外UI显示事件
- 处理局外逻辑

**生命周期**：
```csharp
OnInit()    // 初始化（只调用一次）
OnEnter()   // 进入状态
OnUpdate()  // 每帧更新（可选）
OnLeave()   // 离开状态
OnDestroy() // 销毁（只调用一次）
```

**OnEnter 逻辑**：
1. 触发事件：`OutOfGameEnterEvent`
2. UI 监听事件，自动显示局外UI

**OnLeave 逻辑**：
1. 触发事件：`OutOfGameLeaveEvent`
2. UI 监听事件，自动隐藏局外UI

**设计要点**：
- 不直接操作UI，通过事件通知
- 不保存UI引用，降低耦合
- 场景加载由 Procedure 处理，State 只负责逻辑

### 3. InGameState（局内状态）

**职责**：
- 管理局内子状态机
- 触发局内通用UI显示事件
- 提供子状态切换接口

**核心方法**：
```csharp
// 切换到探索状态
public void SwitchToExploration()

// 切换到战斗状态
public void SwitchToCombat()

// 获取当前子状态
public InGameStateType CurrentSubState
```

**OnEnter 逻辑**：
1. 创建子状态机
2. 注册子状态（Exploration、Combat）
3. 启动子状态机，默认进入 Exploration
4. 触发事件：`InGameEnterEvent`

**OnLeave 逻辑**：
1. 停止子状态机
2. 触发事件：`InGameLeaveEvent`

**设计要点**：
- 子状态机在 OnEnter 时创建，OnLeave 时销毁
- 不缓存子状态机，保持轻量
- 子状态切换通过公共方法暴露

### 4. ExplorationState（探索状态）

**职责**：
- 启用第三人称控制器
- 触发探索UI显示事件
- 处理探索逻辑

**OnEnter 逻辑**：
1. 启用 PlayerController
2. 启用 PlayerSkillManager
3. 触发事件：`ExplorationEnterEvent`
4. 锁定鼠标

**OnLeave 逻辑**：
1. 禁用 PlayerController
2. 触发事件：`ExplorationLeaveEvent`

**设计要点**：
- 直接操作 PlayerController，不额外封装
- 通过 `PlayerInputManager.SetEnable()` 控制输入
- 不管理UI，UI自己监听事件

[↑ 返回目录](#目录)

---

### 5. CombatState（战斗状态）

**职责**：
- 启用战斗系统
- 触发战斗UI显示事件
- 处理战斗逻辑

**OnEnter 逻辑**：
1. 禁用 PlayerController
2. 启用 CombatManager（战斗管理器）
3. 触发事件：`CombatEnterEvent`
4. 初始化战斗数据

**OnLeave 逻辑**：
1. 禁用 CombatManager
2. 触发事件：`CombatLeaveEvent`
3. 清理战斗数据

**设计要点**：
- 战斗系统独立管理，State 只负责启用/禁用
- 战斗结算由 CombatManager 处理
- 战斗结束后触发事件，State 监听事件切换回探索

## 四、事件系统设计

### 1. 事件定义

```csharp
// 局外进入事件
public class OutOfGameEnterEventArgs : GameEventArgs

// 局外离开事件
public class OutOfGameLeaveEventArgs : GameEventArgs

// 局内进入事件
public class InGameEnterEventArgs : GameEventArgs

// 局内离开事件
public class InGameLeaveEventArgs : GameEventArgs

// 探索进入事件
public class ExplorationEnterEventArgs : GameEventArgs

// 探索离开事件
public class ExplorationLeaveEventArgs : GameEventArgs

// 战斗进入事件
public class CombatEnterEventArgs : GameEventArgs

// 战斗离开事件
public class CombatLeaveEventArgs : GameEventArgs

// 战斗结束事件
public class CombatEndEventArgs : GameEventArgs
{
    public bool IsVictory;  // 是否胜利
}
```

### 2. 事件流程

**局外 → 局内（探索）**：
```
OutOfGameLeaveEvent → InGameEnterEvent → ExplorationEnterEvent
```

**探索 → 战斗**：
```
ExplorationLeaveEvent → CombatEnterEvent
```

**战斗 → 探索**：
```
CombatEndEvent → CombatLeaveEvent → ExplorationEnterEvent
```

**局内 → 局外**：
```
ExplorationLeaveEvent → InGameLeaveEvent → OutOfGameEnterEvent
```

## 五、UI 管理设计

### 1. UI 显示逻辑

**原则**：
- UI 不由 State 直接管理
- UI 监听状态事件，自动显示/隐藏
- UI 之间不互相引用

### 2. UI 监听事件

**局外UI**：
- 监听 `OutOfGameEnterEvent` → 显示
- 监听 `OutOfGameLeaveEvent` → 隐藏

**局内通用UI**（HP/MP、Buff、货币）：
- 监听 `InGameEnterEvent` → 显示
- 监听 `InGameLeaveEvent` → 隐藏

**探索UI**（快捷技能栏、星盘）：
- 监听 `ExplorationEnterEvent` → 显示
- 监听 `ExplorationLeaveEvent` → 隐藏

**战斗UI**（召唤师状态、手牌、技能）：
- 监听 `CombatEnterEvent` → 显示
- 监听 `CombatLeaveEvent` → 隐藏

### 3. UI 基类设计

```csharp
// UI 基类（已有 UIFormBase）
public abstract class StateAwareUIForm : UIFormBase
{
    // 在 OnOpen 时订阅事件
    protected override void OnOpen(object userData)
    {
        SubscribeEvents();
    }

    // 在 OnClose 时取消订阅
    protected override void OnClose(bool isShutdown, object userData)
    {
        UnsubscribeEvents();
    }

    // 子类实现事件订阅
    protected abstract void SubscribeEvents();
    protected abstract void UnsubscribeEvents();
}
```

## 六、玩家控制器切换

### 1. 探索模式

**启用**：
- `PlayerController.enabled = true`
- `PlayerInputManager.SetEnable(true)`
- `PlayerSkillManager.enabled = true`

**禁用**：
- `PlayerController.enabled = false`
- `PlayerInputManager.SetEnable(false)`

### 2. 战斗模式

**启用**：
- `CombatManager.enabled = true`
- `PlayerInputManager.SetEnable(true)`（战斗也需要输入）

**禁用**：
- `CombatManager.enabled = false`

### 3. 设计要点

- 不创建新的控制器，直接启用/禁用现有组件
- 输入系统共用，通过 `SetEnable()` 控制
- 避免重复创建和销毁，提高性能

## 八、状态切换流程图

### 1. 局外 → 局内（探索）

```
用户点击"开始游戏"
    ↓
GameStateManager.SwitchToInGame()
    ↓
OutOfGameState.OnLeave()
    - 触发 OutOfGameLeaveEvent
    - 局外UI自动隐藏
    ↓
InGameState.OnEnter()
    - 创建子状态机
    - 启动子状态机（进入 ExplorationState）
    - 触发 InGameEnterEvent
    - 局内通用UI自动显示
    ↓
ExplorationState.OnEnter()
    - 启用 PlayerController
    - 启用 PlayerSkillManager
    - 触发 ExplorationEnterEvent
    - 探索UI自动显示
    - 锁定鼠标
```

### 2. 探索 → 战斗

```
触发战斗事件（如遇敌）
    ↓
InGameState.SwitchToCombat()
    ↓
ExplorationState.OnLeave()
    - 禁用 PlayerController
    - 触发 ExplorationLeaveEvent
    - 探索UI自动隐藏
    ↓
CombatState.OnEnter()
    - 启用 CombatManager
    - 触发 CombatEnterEvent
    - 战斗UI自动显示
    - 初始化战斗数据
```

### 3. 战斗 → 探索

```
战斗结束
    ↓
CombatManager 触发 CombatEndEvent
    ↓
InGameState 监听到事件，调用 SwitchToExploration()
    ↓
CombatState.OnLeave()
    - 禁用 CombatManager
    - 触发 CombatLeaveEvent
    - 战斗UI自动隐藏
    - 清理战斗数据
    ↓
ExplorationState.OnEnter()
    - 启用 PlayerController
    - 启用 PlayerSkillManager
    - 触发 ExplorationEnterEvent
    - 探索UI自动显示
```

### 4. 局内 → 局外

```
用户选择"返回主菜单"
    ↓
GameStateManager.SwitchToOutOfGame()
    ↓
ExplorationState.OnLeave()
    - 禁用 PlayerController
    - 触发 ExplorationLeaveEvent
    - 探索UI自动隐藏
    ↓
InGameState.OnLeave()
    - 停止子状态机
    - 触发 InGameLeaveEvent
    - 局内通用UI自动隐藏
    ↓
OutOfGameState.OnEnter()
    - 触发 OutOfGameEnterEvent
    - 局外UI自动显示
```

## 九、关键设计决策

### 1. 为什么不额外封装 FSM？

**原因**：
- GF 的 FSM 已经足够强大和灵活
- 额外封装会增加复杂度和维护成本
- 直接使用 GF FSM 更轻量，性能更好

### 2. 为什么使用事件而不是直接引用？

**原因**：
- 降低耦合，State 不需要知道 UI 的存在
- UI 可以独立开发和测试
- 方便后续扩展（如添加音效、特效等）

### 3. 为什么不在 State 中管理 UI？

**原因**：
- State 职责单一，只负责逻辑状态
- UI 自己监听事件，自己管理生命周期
- 避免 State 持有大量 UI 引用

### 4. 为什么不做复杂的状态校验？

**原因**：
- GF 的 FSM 已经处理了状态切换的合法性
- 过多校验会增加代码复杂度
- 信任框架，减少冗余代码

### 5. 为什么子状态机在 OnEnter 时创建？

**原因**：
- 保持轻量，不在局外时占用资源
- 避免状态机长期存在导致的内存占用
- 每次进入局内都是全新的状态机，避免状态残留

## 十、后续扩展

### 1. 新增状态

如果需要新增状态（如"剧情状态"、"PVP状态"）：
1. 创建新的 `FsmState` 子类
2. 在状态机中注册
3. 实现 OnEnter/OnLeave 逻辑
4. 定义对应的事件

### 2. 新增 UI

如果需要新增 UI：
1. 继承 `StateAwareUIForm`
2. 实现事件订阅逻辑
3. 在对应事件中显示/隐藏

### 3. 配置化

后续可以将状态切换规则配置化：
- 定义状态切换配置表
- 在运行时读取配置
- 动态注册状态和切换规则

### 4. 状态数据持久化

如果需要保存状态数据：
- 在 OnLeave 时保存关键数据
- 使用 GF 的 `DataNode` 或 `Setting` 系统
- 在 OnEnter 时恢复数据

## 十一、性能优化建议

### 1. 避免频繁切换

- 状态切换有一定开销（事件触发、UI显示/隐藏）
- 避免在短时间内频繁切换状态
- 如果需要临时暂停，使用暂停系统而不是切换状态

### 2. UI 预加载

- 常用UI可以预加载，避免切换时卡顿
- 使用 GF 的 UI 预加载功能
- 在 Procedure 中预加载关键UI

### 3. 事件订阅管理

- UI 在 OnClose 时必须取消订阅事件
- 避免事件监听器泄漏
- 使用 `StateAwareUIForm` 基类统一管理

### 4. 状态机缓存

- 主状态机全局唯一，不销毁
- 子状态机在局内状态时存在，局外时销毁
- 平衡内存占用和性能

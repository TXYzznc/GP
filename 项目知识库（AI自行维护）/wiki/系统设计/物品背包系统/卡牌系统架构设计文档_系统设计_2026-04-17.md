# 卡牌系统架构设计文档

## 概述

本文档描述了新增的扇形卡牌排列系统的整体架构、设计原则、核心算法和交互流程。

---

## 问题陈述

### 原有设计的局限

原方案使用 Unity 的 LayoutGroup 组件自动排列卡牌：
- LayoutGroup 每帧强制重置子物体位置，导致 DOTween 动画被覆盖，动效失效
- 无法实现扇形排列（只能水平/垂直）
- 无法实现拖拽时其他卡让位的动效
- 补位动画需要手动管理，难以协调

### 解决方案

新增 **CardSlotContainer** 脚本完全接管卡牌排列和动效，移除 LayoutGroup，实现：
1. ✅ 扇形排列算法
2. ✅ 进场动画（滑入）
3. ✅ 补位动画（平滑滑动）
4. ✅ 拖拽让位（临时偏移）
5. ✅ 悬停上移（基准坐标叠加）

## 核心算法详解

### 扇形排列算法

#### 参数定义

```csharp
float radius          = 800f;   // 圆弧半径
float maxFanAngle     = 50f;    // 最大展开角度（度）
Vector2 center        = (...);  // 圆心（容器坐标系）
```

#### 计算步骤

给定 N 张卡牌，计算第 i 张卡的位置：

```
1. 将角度范围分成 N 等份
   angleStep = maxFanAngle / (N - 1)
   
2. 计算第 i 张卡的角度
   halfRange = maxFanAngle / 2
   angle[i] = -halfRange + i * angleStep
   
3. 计算圆弧上的坐标
   x[i] = sin(angle[i]) * radius
   y[i] = cos(angle[i]) * radius - radius
   pos[i] = center + (x[i], y[i])
   
4. 计算卡牌旋转（与圆弧切线对齐）
   rotZ[i] = -angle[i]
```

#### 几何示意

```
示例：3 张卡，maxFanAngle = 50°

角度分布：
  卡0: -25°    卡1: 0°    卡2: +25°
  
圆弧排列：
        ┌─────────┐
      [0]       [1]       [2]
    ╱             │         ╲
   圆心 ─────────────────────── 水平线
```

#### 实现代码位置

- `CardSlotContainer.cs` → `CalculateFanPositions()`
- 返回值：`FanTransform[]` 数组，包含每张卡的目标位置、旋转、缩放

## 动效系统

### 1. 进场动画 (AddCardAsync)

**触发**：新卡加入容器
**时长**：0.35 秒
**动画**：
- 起点：Y-200（容器底部）
- 终点：计算的扇形位置
- 淡入：透明度 0 → 1
- 移动：线性插值到目标位置
- 旋转：0 → 目标角度

**实现**：`PlayCardEnterAnimationAsync()`

```csharp
// 伪代码
sequence {
    Fade(0 → 1, 0.35s)
    + Move(startPos → targetPos, 0.35s, OutQuad)
    + Rotate(0 → rotZ, 0.35s, OutQuad)
}
await sequence.AsyncWaitForCompletion()
```

### 2. 补位动画 (RearrangeAsync)

**触发**：移除卡或拖拽结束
**时长**：0.25 秒
**动画**：所有卡牌平滑滑动到新位置
**缓动**：OutCubic（快速减速）

**实现**：
- 计算所有卡的新位置（不包括拖拽卡）
- 遍历每张卡，并行执行 DOTween
- 等待所有动画完成

```csharp
// 伪代码
var fanPos = CalculateFanPositions()
for each card in m_Cards:
    if card != dragCard:
        sequence.Join(
            card.DOAnchorPos(fanPos[i], 0.25s, OutCubic)
            + card.DORotate(rotZ[i], 0.25s, OutCubic)
        )
```

### 3. 悬停动画 (OnPointerEnter/Exit)

**触发**：鼠标进入/离开卡牌
**在基准坐标基础上**：
- Y 偏移：+30 像素（向上）
- 缩放：1.0 → 1.05

**关键设计**：
- 悬停动画不改变 Container 管理的基准坐标
- 只是在 CardSlotItem 本地叠加临时偏移
- 拖拽或其他事件触发时自动恢复

**实现**：
```csharp
// OnPointerEnter
itemRect.DOAnchorPos(m_BasePos + (0, 30), 0.2s)
btnTransform.DOScale(1.05, 0.2s)

// OnPointerExit
itemRect.DOAnchorPos(m_BasePos, 0.2s)
btnTransform.DOScale(1.0, 0.2s)
```

### 4. 拖拽让位

**触发**：`OnCardDrag()` 期间
**逻辑**：
- 根据鼠标 X 坐标计算卡牌应该插入的位置 (insertIndex)
- 左侧卡向左偏移 cardWidth * 0.5f
- 右侧卡向右偏移 cardWidth * 0.5f
- 只是临时偏移，不改变基准坐标

**实现**：`UpdateDragOffsets()`
```csharp
if dragIndex < insertIndex:
    // 拖往右，中间卡左移
    for i in (dragIndex+1, insertIndex):
        offset[i] = -cardWidth * 0.5f
else:
    // 拖往左，中间卡右移
    for i in (insertIndex, dragIndex):
        offset[i] = cardWidth * 0.5f
```

## 交互流程

### 添加卡牌流程

```
CombatUI.CreateCardSlot()
  ↓
CardSlotItem.SetData()
  ↓
await CardSlotContainer.AddCardAsync(card)
  ↓
  ├─ m_Cards.Add(card)
  ├─ await PlayCardEnterAnimationAsync(card)
  │   ├─ 设置起点 (Y-200)
  │   ├─ 计算目标位置 (FanTransform)
  │   ├─ 调用 card.SetBaseFanTransform() 设置基准
  │   └─ 播放淡入+移动+旋转动画
  ├─ await RearrangeAsync(skipNewCard: true)
  │   ├─ 计算所有卡的新位置
  │   ├─ 对每张卡播放补位动画
  │   └─ 更新基准坐标
  └─ 完成
```

### 移除卡牌流程

```
CardManager.RemoveCard(cardId)
  ↓
CardSlotItem.PlayDestroyAnimationAsync()
  ├─ 淡出 + 缩小 (0.3s)
  └─ Destroy(gameObject)
  ↓
CombatUI.OnCardRemoved()
  ↓
CardSlotContainer.RemoveCard(card)
  ├─ m_Cards.Remove(card)
  ├─ await RearrangeAsync(skipNewCard: false)
  │   ├─ 计算剩余卡的新位置
  │   ├─ 播放补位动画
  │   └─ 更新基准坐标
  └─ 完成
```

### 拖拽交互流程

```
CardSlotItem.OnBeginDrag()
  ├─ m_Container.OnCardBeginDrag(this)
  │   ├─ 记录 m_DragCard = this
  │   └─ 初始化 m_InsertIndex
  └─ CreateDragPreview()

鼠标移动...

CardSlotItem.OnDrag(eventData)
  ├─ 更新拖拽预览位置
  └─ m_Container.OnCardDrag(this, screenPos)
      ├─ 计算插入位置 (CalculateInsertIndex)
      └─ 更新其他卡的临时偏移 (UpdateDragOffsets)
          └─ 驱动 DOTween 临时偏移动画

鼠标释放...

CardSlotItem.OnEndDrag(eventData)
  ├─ 销毁拖拽预览
  ├─ m_Container.OnCardEndDrag(this)
  │   ├─ 清除 m_DragCard
  │   └─ await RearrangeAsync() 恢复正常排列
  └─ 判断释放位置是否有效
      └─ ExecuteCardEffect() 或 ReturnToSlot()
```

## 基准坐标概念

### 目的

为了实现 **叠加式动效**（悬停动画不干扰拖拽，拖拽不干扰补位），引入基准坐标：

```
实际坐标 = 基准坐标 + 悬停偏移 + 拖拽偏移 + ...
```

### 维护职责

| 组件 | 职责 |
|------|------|
| CardSlotContainer | 维护和更新基准坐标（m_BasePos, m_BaseRotZ） |
| CardSlotItem | 在基准坐标基础上叠加临时偏移 |

### 示例

```csharp
// Container 计算目标位置
var target = CalculateFanPositions()[i];  // (500, 100), rotZ: 15°

// 设置卡牌的基准坐标
card.SetBaseFanTransform(this, target.AnchoredPos, target.RotationZ);
// 此时卡牌位置 = (500, 100)

// 用户悬停卡牌
card.OnPointerEnter()
// 临时移动到 (500, 130) = 基准 + (0, 30)
// 注意：基准坐标本身未变，只是显示位置变了

// 用户停止悬停
card.OnPointerExit()
// 恢复到基准 (500, 100)
```

## 性能优化

### 当前优化

1. **并行动画**：使用 DOTween.Sequence.Join() 并行执行多个卡牌的动画
2. **条件计算**：只计算非拖拽卡的位置（includeDragCard 参数）
3. **限流射线检测**：OnDrag 中限制射线检测频率为 0.1 秒

### 可进一步优化的方向

1. **预计算**：缓存常见卡数量的排列参数
2. **对象池**：复用 FanTransform 数组而非每次 new
3. **增量更新**：只重新计算受影响的卡，而非全部
4. **异步计算**：将算法放在后台线程（Unity Job System）

## 常见问题解答

**Q: 为什么不用 GridLayoutGroup 加脚本覆盖？**
A: LayoutGroup 每帧强制刷新，导致脚本设置的位置被覆盖。移除 LayoutGroup 给脚本完全控制权。

**Q: 基准坐标是否需要持久化保存？**
A: 不需要。基准坐标是运行时计算的，每次 RearrangeAsync 都会更新。只有卡牌数据（CardData）需要保存。

**Q: 如何支持多种排列方式（垂直/水平/扇形）？**
A: 在 CardSlotContainer 中增加 `LayoutMode` 枚举，根据模式调用不同的计算算法。

**Q: 拖拽时的碰撞检测怎么做？**
A: 当前通过 `CalculateInsertIndex()` 根据鼠标 X 坐标找最近的卡。可扩展为完整的碰撞盒检测。

## 总结

新系统的核心优势：
1. ✅ 完全的脚本控制，动画不被覆盖
2. ✅ 灵活的排列算法（易于扩展到其他形状）
3. ✅ 清晰的职责划分（Container 管排列，Item 管交互）
4. ✅ 基准坐标的叠加设计，使动效能够复合叠加而不冲突
5. ✅ 异步接口（UniTask），时序清晰可控

这个架构为未来的复杂卡牌交互（如手牌扇形展开、多选、编队等）提供了坚实的基础。

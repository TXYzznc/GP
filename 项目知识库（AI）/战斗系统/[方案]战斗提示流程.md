## 问题分析

### 原始问题
- 战斗提示无法显示，出现 `GameFrameworkException: Event '-753117016' not exists specified handler` 错误
- 错误原因：在没有订阅事件的情况下尝试取消订阅

### 架构问题
发现了更深层的架构问题：
1. **重复逻辑** - GameTestManager、GameProcedure、CombatPreparationState 都在处理战斗提示
2. **事件协调复杂** - 通过 GameplayEventArgs.EnterCombatTipClosed 事件来协调UI和状态切换
3. **职责不清** - 战斗提示应该是战斗准备状态的内置流程，而不是外部协调

## 重构方案

### 核心思路
将战斗提示UI作为战斗准备状态的强绑定组件，完全内置到 CombatPreparationState 中处理。

### 新的流程
```
任何入口 → GameStateManager.SwitchToCombatPreparation() 
         → CombatPreparationState.OnEnter()
         → 显示战斗提示UI → 等待UI动画完成 → 继续战斗准备初始化
```

## 修改文件清单

### 1. GameTestManager.cs
- **修改内容**：简化O键处理，直接调用 `GameStateManager.Instance.SwitchToCombatPreparation()`
- **删除内容**：整个 `#region 战斗进入提示` 区域的事件协调代码
- **修改原因**：移除复杂的事件协调，简化测试逻辑

### 2. CombatPreparationState.cs  
- **修改内容**：在 OnEnter 开始时添加 `ShowEnterCombatTipAndContinue()` 异步方法
- **新增方法**：
  - `ShowEnterCombatTipAndContinue()` - 显示提示UI并等待完成
  - `WaitForUIFormClosed()` - 等待UI窗体关闭
  - `ContinueCombatPreparationInit()` - 继续战斗准备初始化
- **修改原因**：将战斗提示作为战斗准备状态的内置流程

### 3. EnterCombatTip.cs
- **修改内容**：OnClose 方法中移除事件触发逻辑
- **修改原因**：不再需要通过事件通知，状态机直接监听UI生命周期

### 4. GameProcedure.cs
- **修改内容**：简化或删除 EnterCombat 相关的战斗提示处理逻辑
- **删除方法**：OnGameplayEvent、OnEnterCombatTipClosed、StartCombatPreparation
- **修改原因**：消除重复逻辑，统一架构

### 5. GameplayEventArgs.cs (可选)
- **修改内容**：移除 EnterCombatTipClosed 枚举值
- **修改原因**：不再使用该事件

## 技术要点

### 1. 异步等待UI关闭
```csharp
// 使用轮询方式等待UI关闭
private async UniTask WaitForUIFormClosed(int uiFormId)
{
    while (GF.UI.HasUIForm(uiFormId))
    {
        await UniTask.Yield();
    }
    await UniTask.Yield(); // 额外等待确保清理完成
}
```

### 2. 错误处理
- UI打开失败时继续初始化流程，确保游戏不中断
- 使用 try-catch 包装异步操作，处理异常情况

### 3. 日志规范
- 统一使用 `DebugEx.LogModule` 替代 `Log.Info`
- 每个关键步骤都有详细的日志输出

## 优势

### 1. 架构清晰
- 单一职责：CombatPreparationState 负责完整的战斗准备流程
- 内聚性高：战斗提示作为战斗准备的内置步骤

### 2. 维护性好
- 消除重复代码
- 减少事件依赖
- 逻辑集中管理

### 3. 稳定性强
- 完善的异常处理
- 即使UI出现问题也不会中断游戏流程

### 4. 扩展性好
- 可以轻松添加更多的战斗准备步骤
- UI动画可以完全自定义

# 第 2-3 章 相关技术与系统架构

---

## 第 2 章 相关技术与理论基础

### 2.1 游戏引擎与 Unity

Unity 是当前最流行的游戏开发引擎之一，具有跨平台、易用、生态完善等优势。本项目采用 Unity 2022 LTS 版本，主要原因包括：

1. **跨平台支持**：支持 PC、移动、主机等多个平台，便于游戏发行
2. **强大的编辑器**：直观的场景编辑、资源管理、调试工具
3. **丰富的资源库**：Asset Store 提供大量现成资源和工具
4. **活跃的社区**：大量教程、文档、第三方库支持

Unity 的核心概念包括：
- **Scene（场景）**：游戏世界的容器，包含所有游戏对象
- **GameObject（游戏对象）**：场景中的基本单位，可以包含多个组件
- **Component（组件）**：附加到游戏对象上的功能模块
- **Prefab（预制体）**：可复用的游戏对象模板

### 2.2 GameFramework 框架

GameFramework 是一个开源的 Unity 游戏框架，提供了完整的游戏开发基础设施。本项目采用 GameFramework 框架的主要模块包括：

**2.2.1 Procedure（流程管理）**
- 管理游戏的不同流程（启动、菜单、游戏、结束等）
- 提供流程间的切换和通信机制
- 支持流程的初始化、更新、销毁

**2.2.2 FSM（有限状态机）**
- 管理游戏对象的状态
- 支持状态间的转移和事件处理
- 用于角色、敌人、UI 等的状态管理

**2.2.3 Event（事件系统）**
- 提供发布-订阅模式的事件通信
- 支持系统间的解耦通信
- 减少模块间的直接依赖

**2.2.4 Entity（实体系统）**
- 管理游戏中的动态对象（角色、敌人、物品等）
- 支持实体的生命周期管理
- 提供实体的查询和管理接口

**2.2.5 UI（UI 系统）**
- 管理游戏的用户界面
- 支持 UI 的打开、关闭、显示、隐藏
- 提供 UI 的生命周期管理

**2.2.6 Resource（资源管理）**
- 统一管理游戏资源的加载和卸载
- 支持资源的缓存和复用
- 提供资源的异步加载接口

### 2.3 回合制 RPG 游戏设计

回合制 RPG 游戏是一种经典的游戏类型，具有以下特点：

1. **回合制战斗**：玩家和敌人轮流行动，每个回合有固定的行动次数
2. **角色成长**：通过战斗获得经验和装备，提升角色属性
3. **策略性**：玩家需要选择合适的技能和装备，制定战斗策略
4. **可平衡性**：相比实时战斗，更容易进行数值平衡

Clash of Gods 项目融合了多种游戏类型的特点：
- **卡牌系统**：策略卡牌的选择和使用
- **自走棋系统**：棋子的部署和战斗
- **肉鸽系统**：地图探索和随机事件
- **搜打撤系统**：物品搜索、战斗、撤离

### 2.4 系统架构设计模式

在游戏开发中，采用合适的设计模式可以提高代码的可维护性和可扩展性。本项目采用的主要设计模式包括：

**2.4.1 事件驱动架构**
- 系统间通过事件进行通信，而不是直接调用
- 优点：低耦合、高灵活性、易于扩展
- 应用：战斗系统、物品系统、UI 系统等

**2.4.2 单例模式**
- 确保某个类只有一个实例，并提供全局访问点
- 应用：管理器类（CardManager、BuffManager 等）

**2.4.3 工厂模式**
- 通过工厂方法创建对象，而不是直接 new
- 应用：卡牌效果创建、Buff 创建等

**2.4.4 对象池模式**
- 预先创建对象，重复使用而不是频繁创建销毁
- 优点：减少 GC 压力、提高性能
- 应用：特效、UI 元素、敌人等

**2.4.5 观察者模式**
- 对象间的一对多依赖关系
- 应用：事件系统、Buff 系统等

### 2.5 配置表驱动设计

配置表驱动设计是一种将游戏数据从代码中分离出来的方法，具有以下优势：

1. **快速迭代**：修改数据无需重新编译代码
2. **易于维护**：数据和代码分离，便于管理
3. **支持热更新**：可以在游戏运行时更新数据
4. **易于平衡**：游戏设计师可以直接修改配置表进行平衡

**工作流程**：
```
Excel 配置表
    ↓
DataTableGenerator 工具
    ↓
自动生成 .cs 类 + .bytes 二进制文件
    ↓
游戏运行时加载
    ↓
代码中使用配置数据
```

本项目包含 15+ 个配置表，涵盖 Buff、卡牌、物品、敌人、技能等各个方面。

## 第 3 章 系统总体设计与架构

### 3.1 游戏整体架构

Clash of Gods 项目采用分层架构设计，从上到下分为以下几层：

**3.1.1 表现层**
- 负责游戏的视觉和音频表现
- 包括 UI、特效、动画、音效等
- 与玩家交互的直接接口

**3.1.2 逻辑层**
- 负责游戏的核心逻辑
- 包括战斗系统、物品系统、卡牌系统等
- 处理游戏规则和数据变化

**3.1.3 数据层**
- 负责游戏数据的管理和存储
- 包括配置表、存档数据、运行时数据等
- 提供数据的读写接口

**3.1.4 框架层**
- 提供游戏开发的基础设施
- 包括 GameFramework 框架的各个模块
- 支持流程管理、事件系统、资源管理等

### 3.2 核心系统模块划分

游戏包含 9 个核心系统，各系统的职责如下：

| 系统 | 职责 | 关键模块 |
|------|------|---------|
| **战斗系统** | 管理战斗流程和规则 | 棋子、Buff、AI |
| **物品背包系统** | 管理物品和背包 | 容器、拖拽、持久化 |
| **策略卡系统** | 管理卡牌和效果 | 卡牌、效果、UI |
| **技术美术系统** | 管理视觉效果 | Shader、特效、渲染 |
| **探索系统** | 管理地图探索 | 敌人、事件、数据 |
| **棋子系统** | 管理棋子战斗 | 战斗、技能、AI |
| **UI 系统** | 管理用户界面 | 飘字、拖拽、预览 |
| **框架系统** | 提供基础设施 | GF、动画、存档 |
| **配置表系统** | 管理游戏数据 | 数据表、转换工具 |

### 3.3 数据流和事件系统

**3.3.1 数据流设计**

游戏中的数据流向如下：
```
用户输入
    ↓
输入系统（PlayerInputManager）
    ↓
逻辑系统处理
    ↓
数据更新
    ↓
事件发送
    ↓
UI 更新
    ↓
渲染输出
```

**3.3.2 事件系统**

事件系统是游戏中各个模块间通信的核心机制。主要事件包括：

- **战斗事件**：战斗开始、战斗结束、棋子死亡等
- **物品事件**：物品获得、物品使用、物品丢弃等
- **卡牌事件**：卡牌选中、卡牌使用、卡牌销毁等
- **UI 事件**：UI 打开、UI 关闭、按钮点击等

事件系统的优势：
- 低耦合：模块间不需要直接引用
- 高灵活性：易于添加新的事件监听者
- 易于扩展：新系统可以轻松订阅事件

### 3.4 资源管理和加载策略

**3.4.1 资源分类**

游戏资源分为以下几类：
- **模型资源**：角色、敌人、物品等 3D 模型
- **纹理资源**：贴图、UI 图片等
- **音频资源**：背景音乐、音效等
- **配置资源**：配置表、预制体等
- **特效资源**：粒子特效、Shader 等

**3.4.2 加载策略**

采用分层加载策略：
1. **启动加载**：游戏启动时加载必要资源
2. **场景加载**：进入场景时加载场景资源
3. **异步加载**：使用 UniTask 进行异步加载，不阻塞主线程
4. **缓存管理**：使用对象池缓存常用资源

**3.4.3 资源卸载**

- 场景切换时卸载不需要的资源
- 定期检查内存使用，卸载未使用资源
- 支持手动卸载特定资源

### 3.5 热修复支持设计

热修复是指在不重启游戏的情况下更新代码和数据的技术。本项目支持以下热修复方式：

**3.5.1 代码热修复**
- 将热修复代码放在 `Assets/AAAGame/Scripts/` 下
- 这些代码在 `Hotfix.asmdef` 程序集内
- 支持在运行时重新加载

**3.5.2 数据热更新**
- 配置表支持热更新
- 可以在游戏运行时更新配置数据
- 无需重启游戏即可生效

**3.5.3 热修复流程**
```
检测更新
    ↓
下载更新文件
    ↓
验证文件完整性
    ↓
应用更新
    ↓
重新加载资源
    ↓
继续游戏
```

```csharp
// 获取所有ID
List<int> ids = DataTableExtension.GetAllIds<PlayerSkillTable>();

// 获取所有行
List<T> rows = DataTableExtension.GetAllRows<T>();

// 条件筛选
List<T> rows = DataTableExtension.GetRowsWhere<T>(row => row.Phase == 1);

// 获取单行
T row = DataTableExtension.GetRowById<T>(id);

// 检查是否存在
bool exists = DataTableExtension.HasRow<T>(id);

// 获取行数
int count = DataTableExtension.GetRowCount<T>();
```

## 优化清单

### 1. NewGameUI.cs

#### 位置 1：LoadAvailableSummoners() - 第 175-186 行

**当前代码：**
```csharp
var summonerTable = GF.DataTable.GetDataTable<SummonerTable>();
if (summonerTable == null)
{
    Log.Error("SummonerTable 未加载");
    return;
}

// 获取所有阶段1的召唤师（初始可选）
foreach (var summoner in summonerTable.GetAllDataRows())
{
    if (summoner.Phase == 1)
    {
        m_AvailableSummoners.Add(summoner);
    }
}
```

**优化后：**
```csharp
// 使用条件筛选，一行搞定
m_AvailableSummoners = DataTableExtension.GetRowsWhere<SummonerTable>(s => s.Phase == 1);

if (m_AvailableSummoners.Count == 0)
{
    Log.Error("SummonerTable 未加载或没有阶段1的召唤师");
    return;
}
```

**优势：**
- 代码从 12 行减少到 6 行
- 更简洁易读
- 自动处理空表情况

```csharp
// 1. 从 PlayerInitTable 读取初始配置

    return null;
}

{
    Debug.LogError("找不到 PlayerInitTable 的初始配置 (Id=1)");
    return null;
}

{
    Debug.LogError("SummonerTable 未加载");
    return null;

    Debug.LogError($"找不到召唤师配置 (Id={summonerId})");
    return null;
}
```

**优化后：**
```csharp
// 1. 从 PlayerInitTable 读取初始配置

{
    Debug.LogError("找不到 PlayerInitTable 的初始配置 (Id=1)");
    return null;

    Debug.LogError($"找不到召唤师配置 (Id={summonerId})");
    return null;
}
```

**优势：**
- 代码从 30 行减少到 12 行
- 更简洁，减少了中间变量

## 优化统计

### 代码行数减少

| 文件 | 优化前 | 优化后 | 减少 |
|------|--------|--------|------|
| NewGameUI.cs | ~40 行 | ~15 行 | -62% |
| PlayerAccountDataManager.cs | ~80 行 | ~30 行 | -62% |
| PlayerSkillUI.cs | ~8 行 | ~4 行 | -50% |
| PlayerSkillManager.cs | ~8 行 | ~4 行 | -50% |
| 其他文件 | ~30 行 | ~15 行 | -50% |
| **总计** | **~166 行** | **~68 行** | **-59%** |

### 优势总结

1. **代码更简洁** - 平均减少 50-60% 的代码行数
2. **可读性更强** - 意图更明确，减少中间变量
3. **统一风格** - 全项目使用相同的配置表读取方式
4. **减少错误** - 自动处理空表情况，减少空指针异常
5. **易于维护** - 修改配置表读取逻辑只需改一处

## 实施建议

### 优先级

1. **高优先级** - PlayerAccountDataManager.cs（使用频繁，优化效果明显）
2. **中优先级** - NewGameUI.cs, PlayerSkillUI.cs（用户直接交互）
3. **低优先级** - PreloadProcedure.cs, MenuProcedure.cs（启动时执行一次）

### 实施步骤

1. 先添加 `DataTableExtension` 的新方法（已完成）
2. 逐个文件进行优化（建议一次优化一个文件，便于测试）
3. 每次优化后运行游戏测试功能是否正常
4. 提交代码时注明"优化配置表读取方式"

## 注意事项

1. **保持功能一致** - 优化后的代码行为必须与原代码完全一致
2. **测试覆盖** - 每次优化后测试相关功能
3. **逐步推进** - 不要一次性修改所有文件
4. **保留注释** - 重要的业务逻辑注释要保留
